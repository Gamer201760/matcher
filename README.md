# 📖 Правила разработки проекта

## 1. Styleguide (кодстайл)

* Используем **PEP 8** как базовый стандарт.
* Форматирование кода через **Black**.
* Статический анализ: **Ruff** и **mypy**.
* Имена:
  * модули, пакеты, функции, переменные → `snake_case`
  * классы и исключения → `PascalCase`
  * константы → `UPPER_CASE`
* Один модуль = одна главная сущность (если модуль перегружен — дробим).
* Логгирование через `logging.getLogger(__name__)`.

---

## 2. Зависимости направлены внутрь (Dependency Rule)

* **Внешние слои знают про внутренние, но не наоборот**:
  * `entity` не зависит ни от кого
  * `usecase` зависит только от `entity` и интерфейсов репозиториев
  * `adapters` зависят от `usecase` и `repository` (для реализации интерфейсов)
* Любая инфраструктура (БД, API, брокеры) подключается **через интерфейсы**, а не напрямую.

---

## 3. Интерфейсы

* **Интерфейс хранится в том слое, где он используется**, а не там, где реализован.
  * Например: интерфейс `UserRepositoryInterface` хранится в `usecase`, потому что именно бизнес-логика его использует.
  * Реализация (`SQLAlchemyUserRepository`) хранится в `repository`.
---

## 4. Интерфейсы через `Protocol`

* Все интерфейсы пишутся через наследование от `typing.Protocol` (PEP 544).
* Пример:

  ```python
  from typing import Protocol
  from entity.user import User

  class UserRepositoryInterface(Protocol):
      def create(self, user: User) -> str: ...
      def get(self, id: str) -> User: ...
      def update(self, user: User) -> None: ...
      def delete(self, id: str) -> None: ...
  ```
---

## 5. Тестирование

* Юнит-тесты пишутся для:
  * `entity` (валидации, бизнес-инварианты)
  * `usecase` (с моками интерфейсов)
* Интеграционные тесты — для `adapters` (с реальной БД, API и т.д.).
* Именование тестов:
  * файлы → `test_*.py`
  * функции → `test_*`

---

## 6. Конфигурация

* Все настройки (БД, брокеры, API ключи) → через и `dotenv`.

---

## 7. Логгирование
* Единый конфиг логирования (`logging_config.py`).
* Уровни: `DEBUG` для разработки, `INFO`/`WARNING`/`ERROR` для продакшена.
* В логах — только бизнес-события и технические ошибки (никаких паролей и токенов).

---

## 8. Структура проекта

* `entity/` → чистые доменные модели (dataclass)
* `usecase/` → бизнес-правила, сценарии использования
* `repository/` → инфраструктура (БД, API, брокеры)
* `adapters/` → слой доступа из внешнего мира (GRPC, FastAPI)
* `tests/` → юнит и интеграционные тесты
* `config.py` → настройки проекта
* `logging_config.py` → конфиг логирования
